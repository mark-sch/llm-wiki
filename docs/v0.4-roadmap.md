# v0.4.0 — AI & Human Dual-Format Roadmap

**Theme:** Make the llmwiki simultaneously **directly consumable by AI agents** and **pleasant for humans to search and read**.

**Status:** Planning. Most items are low-effort additions on top of v0.1–v0.3 infrastructure — no new runtime deps.

**Why this version:** The wiki already renders beautifully for humans (v0.1). v0.2 adds interactive features. v0.3 adds more input sources. v0.4 closes the loop: make the output directly queryable by *other* AI agents (Claude, ChatGPT, Cursor, Cline, Codex, Gemini) without scraping HTML.

---

## Guiding principles

1. **One build, two audiences.** Every `llmwiki build` should produce both a human website and an AI-consumable corpus from the same source — no separate pipeline.
2. **Plain text is king.** AI agents parse plain text more reliably than HTML. We emit both.
3. **Semantic HTML + schema.org.** For crawlers and AI that *do* parse HTML, tag everything with microdata.
4. **No embeddings.** Karpathy's whole point is that LLM wikis replace RAG. Keep the output structured, not vectorized.
5. **Standards first.** Prefer `llms.txt`, `sitemap.xml`, `robots.txt`, RSS, JSON-LD, OpenAPI — well-known formats every agent already understands.

## 🤖 Part A — AI-consumable improvements (14 items)

### A1. `llms.txt` and `llms-full.txt` at the site root

The [llms.txt spec](https://llmstxt.org) (Sep 2024) is a standard `robots.txt`-style file that tells AI agents where the machine-readable content lives. Becoming the de-facto standard for AI-readable docs sites.

**What we ship:**

- `site/llms.txt` — short index. Title, one-sentence description, links to key pages (home, projects, sessions, per-project overviews).
- `site/llms-full.txt` — flattened plain-text dump of every wiki page, in one file, ordered by project → date. Should be under ~5 MB so Claude and ChatGPT can paste it wholesale.
- Both generated in `build.py` from the same source tree.

### A2. JSON-LD graph export at `site/graph.jsonld`

One file containing the full entity/concept/source graph as [schema.org JSON-LD](https://schema.org/docs/jsonldcontext.jsonld). Uses `Thing`, `CreativeWork`, and `citation` relations so any schema-aware agent can traverse the whole wiki as structured data.

**Example payload shape:**

```json
{
  "@context": "https://schema.org",
  "@graph": [
    {
      "@id": "project/ai-newsletter",
      "@type": "CreativeWork",
      "name": "ai-newsletter",
      "containsPart": [
        {"@id": "session/clever-munching-parnas"}
      ]
    },
    ...
  ]
}
```

### A3. Per-page `.txt` alongside every `.html`

For every `site/sessions/<project>/<slug>.html`, also emit `site/sessions/<project>/<slug>.txt` — the same content but in plain text, no HTML tags, no JS, no CSS. Agents that can't render HTML can fetch the `.txt` directly.

### A4. Per-page `.json` alongside every `.html`

For every session page, also emit a `.json` file with structured fields:

```json
{
  "slug": "curious-parsing-ada",
  "project": "demo-alpha",
  "date": "2026-04-01",
  "model": "claude-sonnet-4-6",
  "started": "2026-04-01T09:15:00+00:00",
  "ended": "2026-04-01T09:42:18+00:00",
  "user_messages": 3,
  "tool_calls": 5,
  "tools_used": ["Bash", "Read", "Write"],
  "summary": "...",
  "turns": [
    {"role": "user", "text": "..."},
    {"role": "assistant", "text": "...", "tool_calls": [...]}
  ],
  "wikilinks_out": ["Quarkus", "Pratiyush"],
  "wikilinks_in": []
}
```

### A5. Sitemap + RSS feeds

- `site/sitemap.xml` — every page with `<lastmod>` from frontmatter `started`/`ended`
- `site/rss.xml` — global feed of the newest 50 sessions
- `site/projects/<project>/rss.xml` — per-project feed
- `site/feeds/latest.atom` — Atom 1.0 equivalent

### A6. `robots.txt` allow-list

Explicit robots.txt that mentions the `llms.txt` location per the spec:

```
User-agent: *
Allow: /
Sitemap: /sitemap.xml

# See /llms.txt for AI-agent-friendly index
# See /llms-full.txt for full flattened content
```

### A7. OpenAPI spec for the MCP server

The v0.2 MCP server exposes 6 tools. v0.4 ships `docs/openapi-mcp.yaml` — an OpenAPI 3.1 spec describing every tool, its arguments, and its response shape. Useful for:

- Non-MCP clients that want to hit the MCP server via HTTP proxy
- Tool discovery
- Automated client-binding generation (Python, TypeScript, Go)

### A8. Schema.org microdata in every HTML page

Sprinkle `itemtype`, `itemprop`, `itemscope` on:

- `<body>` → `Article` or `CreativeWork`
- `<h1>` → `name`
- `.hero-sub` time → `dateCreated`
- `<a>` to entities → `mentions`
- `<code>` → `text`

Crawlers like Google's and Bing's use schema.org microdata for rich snippets. Claude, ChatGPT, and Gemini browsing modes all parse it.

### A9. Machine-readable frontmatter appendix

Every session page already has YAML frontmatter in the source. v0.4 emits a **visible** HTML comment block at the top of the rendered page containing the same frontmatter, so AI agents scraping HTML can parse metadata without needing to fetch the raw `.md`.

```html
<!-- llmwiki:metadata
project: demo-alpha
slug: curious-parsing-ada
date: 2026-04-01
model: claude-sonnet-4-6
tools_used: [Bash, Read, Write]
-->
```

### A10. Consistent URL canonicalisation + `<link rel="canonical">`

Every generated page has a canonical URL tag pointing at the `.html` version. Prevents duplicate indexing when the `.txt` and `.json` siblings exist.

### A11. Per-page cite-ready block

Near the bottom of every session page, add a collapsed `<details>` block with copy-paste-ready citations in 3 formats: plain (markdown link), BibTeX, and APA.

```
> ai-newsletter/curious-parsing-ada. llmwiki, 2026-04-01.
> https://pratiyush.github.io/llm-wiki/sessions/ai-newsletter/curious-parsing-ada.html
```

### A12. Deterministic IDs

Every entity/concept/session gets a stable ID in the format `<project>/<slug>`. These IDs are used in:

- JSON-LD graph (`@id`)
- Wikilink targets
- MCP tool responses
- Sitemap URLs

Never changes once published (even if title or slug display is tweaked).

### A13. MCP tool: `wiki_export`

New MCP tool `wiki_export(format: "md" | "txt" | "jsonld" | "all", output_path?)` that dumps the entire wiki in the requested format. Useful for agents that want to pull the whole wiki into their context in one call.

### A14. `wiki/ai-readme.md` — an AI-specific entry point

A hand-curated (or LLM-generated) page at the top of the wiki that's optimized for an agent's first visit:

```markdown
# This wiki, explained for AI agents

You are reading a Karpathy-style LLM Wiki. To answer questions:

1. Read `wiki/index.md` for the catalog
2. Read `wiki/overview.md` for the living synthesis
3. Use wikilinks `[[Name]]` to traverse entities and concepts
4. Each session source has frontmatter (project, date, model, tools_used)
5. Structured data: `/llms.txt`, `/llms-full.txt`, `/graph.jsonld`
6. Per-page: sibling `.txt` and `.json` files
```

## 👤 Part B — Human-readable improvements (12 items)

### B1. Reading time estimates

Calculate word count per page, emit `X min read` next to the date in the meta strip. Uses 225 wpm standard.

### B2. TOC sidebar on long pages

Sessions with more than 5 headings get a sticky right-side TOC with scroll-spy (active heading highlighted as you read). Toggleable, remembers state per page.

### B3. Related pages panel

At the bottom of every session page, show 3-5 "related" sessions computed from shared entity/concept wikilinks. Purely client-side JS using the existing `search-index.json`.

### B4. Search result snippets with highlights

When Cmd+K shows results, highlight the matched terms in the result body preview with `<mark>`. Pulls a ±30-char window around the first match.

### B5. Breadcrumb scroll-spy

On long session pages, the breadcrumb bar updates as you scroll to show the current section (`Home › Projects › ai-newsletter › clever-munching-parnas › Turn 12`).

### B6. Reading progress persistence

Remember scroll position per page in `localStorage` so reopening the same session returns you where you were reading.

### B7. Filter bar presets

Save the current filter state (project + model + date range + text) as a named preset. Dropdown shows saved presets, one-click to restore.

### B8. Activity heatmap on the home page

Per-project GitHub-style activity heatmap showing session frequency by day. Compact, one row per project. Uses the existing frontmatter `started` dates — no new data.

### B9. Keyboard shortcut: `Shift+?` shows the keyboard shortcut help overlay

Already have `?` in v0.1. v0.4 adds a tooltip that flashes "press ? for shortcuts" on first visit (localStorage-gated, one-time only).

### B10. Mobile bottom bar

On narrow viewports, the top nav collapses and a bottom-anchored bar with Home/Projects/Sessions/Search icons appears. Standard mobile-web pattern.

### B11. Print CSS polish

- Page breaks before `h2`
- No breaks inside `<pre>` blocks
- Hide nav, footer, action buttons, palette, progress bar
- Show URL in footer for every internal link

### B12. Sharable deep-links to any heading

Every `h2`/`h3` already has an anchor (Pygments codehilite extension provides `.headerlink`). v0.4 adds a hover-visible "link" icon next to each heading that copies a canonical URL with the anchor to clipboard.

## 🔀 Part C — Cross-cutting infra (6 items)

### C1. Build outputs manifest at `site/manifest.json`

One file listing every generated output with its SHA-256 hash and byte size. Used by CI to detect drift between builds and by AI agents to verify they have the latest version.

### C2. Build-time link checker

Before writing `site/`, verify every internal link, wikilink, and asset reference resolves. Fail the build on broken links. Currently `/wiki-lint` does this at the wiki layer; v0.4 extends it to the HTML output.

### C3. Deterministic builds

Given the same `raw/` tree, two `build` runs should produce byte-identical `site/` output (modulo timestamps in frontmatter). Enables reliable CI diffs.

### C4. `llmwiki export <format>` CLI subcommand

New top-level subcommand that produces the AI-consumable outputs independently of `build`. Options: `md`, `txt`, `json`, `jsonld`, `llms-txt`, `rss`, `sitemap`, `all`.

### C5. Content-addressable page URLs (optional)

Each page gets a secondary URL at `/p/<sha256-16>.html` that's stable across renames. Canonical URL in the `<link>` points at the semantic slug; the hash URL is a redirect.

### C6. Performance budget enforcement

CI fails if:

- Any single HTML page exceeds 500 KB
- `llms-full.txt` exceeds 10 MB
- Cold build exceeds 15s on the demo corpus

Currently these are documented targets. v0.4 promotes them to hard CI gates.
